# Localization (i18n) Guide

Upcheck ships in **6 languages** — English plus five Indian languages — powered by `i18next` + `react-i18next`. This guide covers adding strings, namespaces, and languages, and the completeness rule that keeps every farmer seeing real translations, never raw keys.

See also: [../ARCHITECTURE.md](../ARCHITECTURE.md) · [../FEATURES.md](../FEATURES.md) · [frontend guide](./frontend.md).

---

## Languages

Defined in `src/i18n/languages.ts`:

| Code | Language | Endonym |
|------|----------|---------|
| `en` | English | English |
| `hi` | Hindi | हिन्दी |
| `ta` | Tamil | தமிழ் |
| `te` | Telugu | తెలుగు |
| `bn` | Bengali | বাংলা |
| `or` | Odia | ଓଡ଼ିଆ |

`en` is both the default (`lng: 'en'`) and the fallback (`fallbackLng: 'en'`). The chosen language is persisted to AsyncStorage under `@upcheck_language` and restored on launch; every `i18n.changeLanguage()` re-persists it. The picker shows `nativeLabel`.

## Structure

```
src/i18n/
  index.ts            # i18next.init — registers resources, fallback, persistence
  languages.ts        # LANGUAGES + LANGUAGE_CODES
  locales/
    en/
      index.ts        # imports every namespace, exports one aggregated object
      common.ts       # each namespace = a file exporting `default { … }`
      auth.ts  home.ts  farms.ts  ponds.ts  logs.ts  engines.ts  …  (19 namespaces)
    hi/  ta/  te/  bn/  or/   # same 19 namespaces + index.ts, one per language
```

- Each **namespace file** exports a plain default object of keys:
  ```ts
  // locales/en/common.ts
  const common = { save: 'Save', cancel: 'Cancel', networkError: 'Cannot reach the server…' };
  export default common;
  ```
- Each language's **`index.ts`** imports all namespaces and re-exports them as one object:
  ```ts
  // locales/en/index.ts
  import common from './common';
  import auth from './auth';
  // …
  export default { common, auth, home, farms, ponds, logs, engines, … };
  ```
- **`i18n/index.ts`** maps each language object to i18next under a single `translation` namespace, so callers use dotted keys — `t('common.save')`, `t('logs.chemical_title')`:
  ```ts
  const resources = { en: { translation: en }, hi: { translation: hi }, /* … */ };
  ```

Namespaces are *logical groups inside one i18next namespace* (`translation`), not separate i18next namespaces — the grouping is purely the nested object shape.

The 19 namespaces: `common`, `history`, `auth`, `home`, `navigation`, `farms`, `ponds`, `pondSetup`, `feedStats`, `cycles`, `logs`, `calculators`, `simulations`, `finance`, `inventory`, `content`, `settings`, `engines`, `members`.

Use it in a component:

```ts
import { useTranslation } from 'react-i18next';
const { t } = useTranslation();
t('common.save');
t('home.quickLogForPond', { pond: pondName });   // interpolation
```

## Add a new string

1. Add the key to the English namespace first — `locales/en/<namespace>.ts`. This is the canonical key set.
2. Add the **same key** to all five other languages: `hi/te/ta/bn/or/<namespace>.ts`.
3. Reference it as `t('<namespace>.<key>')`.

Rules:

- **Placeholders identical across languages** — `{{pondName}}`, `{{count}}` must be spelled the same in every locale (only the surrounding words translate). `escapeValue` is off, so `{{...}}` interpolation is safe.
- **Unit/technical tokens stay verbatim** — `kg`, `pH`, `FCR`, `ppm`, `DO`, `NH₃` are not translated; translate the words around them.
- Don't hard-code English in a screen "for now" — it silently bypasses localization for 5 languages.

---

## Add a new namespace

1. Create the file in **every** language folder: `locales/<lang>/<newns>.ts`, each exporting `default { … }`.
2. Wire it into each language's `index.ts` (import + add to the exported object) — for `en` and all five others.
3. Use `t('<newns>.<key>')`.

Miss a language's `index.ts` and that language loses the whole namespace (falls back to English at runtime, but the invariant below is broken).

---

## Add a new language

1. Add it to `LANGUAGES` in `languages.ts` (`code`, `label`, `nativeLabel`). `LANGUAGE_CODES` derives automatically and gates the persisted-preference restore.
2. Create `locales/<code>/` with **all 19 namespace files + `index.ts`**, translating every key.
3. Register it in `i18n/index.ts`: `import xx from './locales/xx'` and add `xx: { translation: xx }` to `resources`.

## The completeness invariant

**Every language must contain every `en` key. Coverage is currently 100%** — all 6 locales carry all 19 namespaces (20 files each incl. `index.ts`).

Why it matters: a missing key falls back to English (`fallbackLng: 'en'`), so users never see a raw key like `logs.chemical_title` — but they *do* see an English word dropped into an otherwise-translated screen, which reads as broken. Avoid gaps; treat a fallback as a bug, not a feature.

### Diffing key coverage (transpile-and-flatten)

The locale files are TypeScript (`export default { … }`), so grepping keys is unreliable. To compare coverage, transpile each language's aggregated `index.ts` to JS, import it, and flatten the nested object to dotted key paths — then diff the key sets against `en`:

```bash
# flatten one language to sorted dotted keys, compare to en
npx tsx -e "
  import en from './src/i18n/locales/en';
  import xx from './src/i18n/locales/hi';   // swap the language
  const flat = (o, p='') => Object.entries(o).flatMap(([k,v]) =>
    v && typeof v === 'object' ? flat(v, p+k+'.') : [p+k]);
  const a = new Set(flat(en)), b = new Set(flat(xx));
  console.log('missing in hi:', [...a].filter(k => !b.has(k)));
  console.log('extra in hi:',   [...b].filter(k => !a.has(k)));
"
```

Run it per language in CI or before release; a clean run (no missing keys in any locale) proves the invariant holds.

Translations for all five non-English languages were **recently completed** and are at 100% key coverage, but they are **flagged for native-speaker QA** — the coverage check proves *presence*, not *quality*.
